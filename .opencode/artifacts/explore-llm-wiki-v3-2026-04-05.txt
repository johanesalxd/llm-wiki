### 7.4 lint.py LOG_ENTRY_RE pattern vs SKILL.md log format

SKILL.md log format (ingest, line 83): `## [YYYY-MM-DD] ingest | <source-title> | <source-type> | <slug>`
SKILL.md log format (compile, line 124): `## [YYYY-MM-DD] compile | <slug> | files updated: memory/project-foo.md, ...`

lint.py LOG_ENTRY_RE (line 230–232):
  r"^## \[\d{4}-\d{2}-\d{2}\] (?:ingest|compile) \| .+ \| .+ \| (.+)$"

This regex expects 4 pipe-delimited fields. The ingest log format has 4 fields: [date], ingest, title, source-type, slug — the regex captures the last field (slug). This is correct for ingest entries.

The compile log format has a different structure: `## [YYYY-MM-DD] compile | <slug> | files updated: <list>` — only 3 pipe-delimited fields (date context, compile, slug, files updated as one field). The LOG_ENTRY_RE requires 4 fields (`.+ \| .+ \| (.+)$`). Compile log entries will NOT match the regex and will be silently skipped in log integrity checks.

This means: the log integrity check (Check 5) only validates ingest log entries, not compile log entries. Compile log entries will never trigger an orphaned log entry warning, even if the corresponding slug file is deleted. This is a lint.py correctness gap — not critical (compile logs are less critical than ingest logs for file-existence checks), but it's a discrepancy between what the check is documented to do and what it actually does.
File: scripts/lint.py, line 230–232; SKILL.md lines 83, 124.
Severity: Low (data integrity gap, not a blocking issue).
